The Way of

Why we map journeys, and how a designer turns a promise into a contract we can prove.

dō / michi — 道 — “the way; the path one walks”

道
The thesis

A journey is a 道 — the whole path a person walks, not the stones underfoot.

Stories pave the stones. A journey is the walking — start to finish, on real ground. We build the stones; we are paid for the walk.

旅
Matsuo Bashō · 奥の細道 · 1689

“The months and days are the travellers of eternity. The years that come and go are travellers too.”

Bashō walked the narrow road and wrote down each station. A journey contract is that travel diary: the path, recorded, so anyone can walk it again and know they arrived.

Oku no Hosomichi — The Narrow Road to the Deep North

間
間 · ma — the meaningful interval

The journey lives in the gaps between the steps.

ma is the space between — and it is where a real person gets stranded: a missing redirect, a balance that reads zero, a record that never reaches the next screen. Button A passes. Button B passes. The human is stuck in the ma. Journeys test the walking, not the stones.

完
Definition of Done

“Done” is not “the button works.” Done is: a real human finished, on real data.

  • A feature ships when its CRITICAL journeys are green against the real backend.
  • Real session, not mocks. Every assertion checks a real value — a record, a status, a number.
  • The list of what must pass lives in CONTRACT_INDEX.yml → definition_of_done.
型
型 · kata — the perfected form

The journey contract is a kata: the agreed form you must be able to perform.

In Noh and the martial arts, the kata is the form rehearsed until it can be performed flawlessly, on demand. The journey_*.yml is our kata — written down, registered, and demonstrated green before we claim mastery.

Zeami, Fūshikaden · the Noh treatises

図
What we must have, to build

The wall first. Then agreement. Then — and only then — the CSV.

You digitize only what has been seen whole and agreed. The CSV is not the start; it is the moment the journey goes digital.

① Map on the wall · Figma/FigJam→ ② Agree the set + tiers→ ③ Convert to CSV→ spec → yaml → e2e
守破離
守破離 · shu-ha-ri — the arc of mastery

How a journey grows up.

守 Shu — keep the form

Map it on the wall, agree it, write it into the CSV. Follow the path exactly as drawn.

破 Ha — embody it

Spec → machine-readable contract → real-backend e2e. The form becomes enforceable; it goes green.

離 Ri — let it live

The journey is alive and visible; it changes only through the ritual, never by editing the YAML in the dark.

Sen no Rikyū (tea) · later the martial Ways

誠
誠 · makoto — sincerity, truth

Prove it on real ground, or you have proved nothing.

  • Real backend — a seeded real user, not a mocked API pretending.
  • Oracle-anchored — assert the real value; where a wrong answer exists, assert it is not shown.
  • A green-under-mocks journey is tatemae — the polite façade. We require makoto.

“Perceive that which cannot be seen with the eye.”

Miyamoto Musashi · 五輪書 The Book of Five Rings · 1645

見
Keeping journeys visible

The YAML is the destination, never the origin — so it never goes obscure.

Truth flows one way: wall → CSV → spec → YAML → e2e. Nobody invents a journey inside the YAML. Three views, kept in sync, share one vocabulary — the journey_id:

  • The wall (Figma) — the human map, every node tagged with its journey id.
  • The board — a dashboard generated from CONTRACT_INDEX: passing / failing / stub / last-verified, visible at a glance.
  • The YAML — the enforceable detail, linked both ways.
改善
改善 · kaizen — changing a journey

Change it the way it was born: from the wall, through agreement, down the ladder.

Refinement

Add a step, tighten a check. Same id. Update spec + YAML + e2e in one PR; re-prove.

Material change

The promise changes (outcome / tier). Back to the wall, re-agree. Shared rule? An ADR.

Supersession

Replaced. Retire the old id (status: retired), create a new one. Never mutate a passing meaning in the dark.

Always

Reset the proof: status: not_tested until green again. The id is stable; its meaning is governed.

契
How this relates to specflow

Journeys are specflow — at the altitude of the whole task.

Feature contract — the inner loop

feature_*.yml asserts an invariant (“only this service writes X”), enforced by fast tests/contracts/*.

Journey contract — the outer loop

journey_*.yml asserts a promise (a human finishes the task), enforced by real-backend e2e.

One registry, one gate (specflow:verify + test:contracts), one creed: contract before code, verify before done. A journey’s covers_reqs ties it to the feature requirements it proves.

筆
For the designer — your hand on the brush

Your job is Step 1: one CSV row per journey. Two columns carry the weight.

success_outcome — the oracle

One checkable fact: a record, a status, a number. “Status reads ‘Filed’ and a NOL row exists.” Never a feeling.

parity_baseline — sorts the tier

How a person does this today (adjuster / spreadsheet / portal). Must-match → CRITICAL. Can’t-do-today → DIFFERENTIATOR.

Describe intent, not UI. Engineering names the selectors. Get these two right and the contract almost writes itself.
